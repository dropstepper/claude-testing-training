# WSL2 の nginx を Postman でテストする

**所要時間：** 20〜30 分  
**対象：** 「WSL2 で Linux コマンドを体験する」を完了した受講者  
**前提：**
- VS Code のターミナルで `wsl` コマンドを実行して Ubuntu のシェルに切り替えられること
- 前の演習（WSL2 で Linux コマンドを体験する）で nginx がインストール済みであること
- VS Code の Postman 拡張機能にサインイン済みであること

---

## この課題の目的

課題3 では ATRS という実際のアプリの API をテストしました。  
この課題では nginx という「シンプルな Web サーバー」を相手に、HTTP 通信の基礎を確認します。  
ゴールは「リクエストを送るとどんなレスポンスが返るか」をボディだけでなくヘッダーまで含めて読めるようになることです。

---

## ① nginx の起動と疎通確認

### 手順1：nginx を起動する

VS Code のターミナルで以下のコマンドを実行して Ubuntu のシェルに切り替えます。

```
wsl
```

プロンプトが切り替わったら、続けて nginx を起動します。

```
sudo systemctl start nginx
```

**確認チェックリスト：**

- [ ] エラーなく完了した

> **トラブルシューティング**
> `System has not been booted with systemd` というエラーが表示された場合は、`sudo service nginx start` を代わりに使用してください。

---

### 手順2：コマンドラインで疎通確認する

```
curl http://localhost
```

**確認チェックリスト：**

- [ ] HTML のコードが出力された
- [ ] `Welcome to nginx!` という文字列が含まれていた

> **メモ：** `curl` は HTTP リクエストをターミナルから送るコマンドです。Postman の「ターミナル版」とイメージしてください。Postman を開く前にまずコマンドラインで疎通を確認するのは、現場でもよく使う手順です。

---

## ② Postman で GET リクエストを送る

### 手順3：GET リクエストを送信する

```
HTTP メソッド：GET
URL：http://localhost
```

**操作手順：**

1. VS Code の Postman を開く
2. HTTP メソッドのドロップダウンから **GET** を選択する
3. URL フィールドに `http://localhost` を入力する
4. **[Send]** をクリックする

**確認チェックリスト：**

- [ ] ステータスコードが **200** であることを確認した
- [ ] レスポンスボディに `Welcome to nginx!` が含まれていることを確認した

> **気づきポイント：** curl と Postman は同じ `http://localhost` に対して同じリクエストを送っています。ツールが違っても HTTP の動作は変わりません。

---

## ③ レスポンスヘッダーを読む

HTTP のレスポンスには「ボディ」（本文）のほかに「ヘッダー」（付帯情報）が含まれます。  
課題3 では主にボディを確認しましたが、この課題ではヘッダーに注目します。

### 手順4：Headers タブを確認する

Postman のレスポンスエリアで **[Headers]** タブをクリックしてください。

**確認チェックリスト：**

- [ ] `Server` ヘッダーに `nginx/（バージョン番号）` が表示されていた
- [ ] `Content-Type` ヘッダーに `text/html` が含まれていた

> **気づきポイント：** `Server` ヘッダーはレスポンスを返したソフトウェアの名前を示します。`Content-Type` はレスポンスの種類を示し、`text/html`（HTML 文書）と `application/json`（JSON データ）ではボディの構造がまったく異なります。課題3 の ATRS API は `application/json` を返していました。

---

## ④ サービスを停止してエラーを確認する

### 手順5：nginx を停止する

```
sudo systemctl stop nginx
```

**確認チェックリスト：**

- [ ] エラーなく完了した

---

### 手順6：停止後にリクエストを送る

Postman から再度 `http://localhost` に **[Send]** してください。

**確認チェックリスト：**

- [ ] Postman に接続エラーが表示された（ステータスコードは返らない）

> **気づきポイント：** 「サービスを止めるとリクエストが失敗する」という動作を意図的に確認しておくことで、「API テストが失敗した原因がサーバー停止なのか、それともアプリのバグなのか」を切り分ける感覚が身につきます。テストの失敗原因を特定するとき、まずサービスが起動しているかを確認するのは基本の手順です。
