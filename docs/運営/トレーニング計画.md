# Claude Code × テスト設計 トレーニング 計画

本ドキュメントは、テストエンジニアがATRS（航空券予約システム）を題材に、  
AI（Claude Code）を活用したテスト設計スキルを実践的に習得するためのトレーニング計画です。

---

## 対象者プロフィール

- テストエンジニア（非情報系出身者も含む）
- JSTQB / ISO 29119 の基礎知識の習得が必要
- GitHub / Playwright / Postman の実務経験が浅い
- 開発環境（VS Code, Claude Code）は設定済み

---

## 使用環境の全体像

```
┌─────────────────────────────┐    ┌─────────────────────────────┐
│     AWS（クラウド）           │    │    受講者のPC（ローカル）      │
│     t2.micro / 無料枠        │    │    Windows                  │
│                             │    │                             │
│  ・Ubuntu 24.04 LTS         │    │  ・ATRS（正常版 / バグ版）    │
│  ・nginx（練習用 Web サーバー）│    │  ・VS Code + Claude Code    │
│                             │    │  ・Postman                  │
│  → Linux コマンド練習         │    │  ・Playwright               │
│  → Postman の基礎練習         │    │  ・Git / GitHub             │
└─────────────────────────────┘    └─────────────────────────────┘
        ↑ メンターが事前に構築・準備            ↑ 研修の主戦場
```

| 環境 | 用途 | 備考 |
|---|---|---|
| AWS t2.micro（Ubuntu + nginx） | Linux コマンド練習 / Postman 基礎 | メンターが事前構築。無料枠で運用 |
| ローカル ATRS（正常版 Port:8082） | 期待動作確認 / Postman 実践 / Playwright | 受講者の PC で起動 |
| ローカル ATRS（バグ版 Port:8083） | 不具合発見演習 | 同上 |

## AI・ツール利用における情報セキュリティ

**一度送信した情報は削除できない。**  
AI ツールへの入力・GitHub への push・画面共有のいずれも、操作した瞬間に情報はサーバーに送信され、自分では制御できなくなる。  
研修中だけでなく、案件参画後も同様に適用される。

### 判断基準

> **「これは公開していい情報か？」—— 一秒で判断できないなら入力しない。**

これだけ守ればほとんどのリスクは防げる。

### 入力してはいけない情報

| カテゴリ | 具体例 |
|---|---|
| 個人情報 | 氏名・住所・電話番号・メールアドレス・マイナンバー等 |
| 業務上の機密情報 | 顧客情報・社内ドキュメント・未公開仕様・売上データ等 |
| 認証情報 | パスワード・API キー・トークン・SSH 秘密鍵等 |
| 参画先の内部情報 | 案件のソースコード・DB スキーマ・インフラ構成等 |

**なぜ取り返しがつかないか：**  
「案件の仕様書をそのまま貼って質問した」という行為が、顧客との秘密保持契約（NDA）違反になり得る。Claude Code / ChatGPT 等に入力したデータはプロバイダのサーバーに送信され、ツールやプランによっては学習データに使われる。削除を要求しても完全な消去は保証されない。  
エラーログや設定ファイルも同様に注意が必要。スタックトレースにはパス・バージョン・環境情報が含まれることが多い。**最小限の情報に絞ってから質問する。**

### Git / GitHub への push 時の注意

> **実際に起きること：** API キーを含むファイルを GitHub の public リポジトリに push した直後、自動スキャンボットに検知され数分〜数時間以内に悪用されるケースが報告されている。AWS のアクセスキーが漏洩した場合、一晩で数十万円の請求が発生した事例もある。

- push 前に `git diff` で差分を目視確認する
- `.env`・`*.pem`・`secrets/` は必ず `.gitignore` に登録する
- **誤って push してしまった場合：** まずキーを無効化（ローテーション）し、すぐにメンターに報告する。履歴から削除しても漏洩は取り消せないと心得る

### 案件参画後のツール利用

- **参画初日**に「AI ツールの利用可否」をチームリーダーまたは PM に確認する。承認なく導入しない
- 個人の SaaS アカウント（無料プラン）を業務に使わない。無料プランは学習データへの利用を許可している場合が多い
- **画面共有・スクリーンショット時も同様。** 顧客データ・社内情報が映り込んでいないか確認してから共有する

## 講師（メンター）の事前準備

研修開始前に以下を完了させておく。受講者の学習時間を確保するため、環境構築は講師側で完結させる。

| 作業 | 内容 |
|---|---|
| AWS EC2 起動 | Ubuntu 24.04 LTS / t2.micro / 無料枠 |
| nginx インストール | `sudo apt install nginx` で起動確認 |
| セキュリティグループ設定 | SSH（ポート22）・HTTP（ポート80）を開放 |
| 受講者用 Linux ユーザー作成 | `sudo adduser <ユーザー名>` で個別アカウントを発行 |
| SSH 接続情報の準備 | IPアドレス・ユーザー名・接続手順を文書化 |
| ローカル ATRS の起動確認 | 正常版・バグ版ともに動作確認済みの状態で渡す |
| AWS 課金アラート設定 | AWS Budgets で月 $1 のアラートを設定（保険） |

> **後片付け（研修終了後）：** インスタンスを「停止（Stop）」ではなく **「終了（Terminate）」** すること。これにより仮想サーバーとディスクの両方が消去され、課金リスクがゼロになる。

## トレーニングフェーズ

### Phase 1：基礎知識の習得（面談対策）

**目的：** テストエンジニアとしての共通言語を身につける。AIへの正確な指示にも直結する。

- Verification（検証）と Validation（妥当性確認）の違い
- 静的テスト / 動的テスト の違い
- ISO/IEC/IEEE 29119 の基本構造の理解
- JSTQB FL の主要テスト技法の概要
  - 同値分割法 / 境界値分析 / デシジョンテーブルテスト

> **参考：** `docs/AI x テスト自動化 アーキテクチャと運用プロセス.md` セクション1「面談対策」

---

### Phase 2：テストケース作成 × コンテキストエンジニアリング ★最優先★

**目的：** 仕様書を Claude Code に渡してテストケースを生成し、  
プロンプトを改善しながらアウトプットの質を上げる「コンテキストエンジニアリング」を体験する。

#### Step 1：仕様書を読み込ませてテストケースを生成する
1. Claude Code に `ATRS 要件定義書.pdf` / `ATRS 外部設計書.pdf` を読み込ませる
2. 対象機能（例：ログイン・航空券検索・予約）を指定して初回テストケースを生成させる
3. 生成結果をレビューし、不足している観点を特定する

#### Step 2：プロンプトを改善して品質を上げる（コンテキストエンジニアリング）
1. 指示に「ISO 29119 / JSTQB の観点でレビューして」を追加する
2. 「同値分割・境界値分析・デシジョンテーブルが適用されているか確認して」と指示する
3. 指示の書き方（渡す情報量・粒度）を変えてアウトプットの違いを観察する

#### Step 3：Mermaid 図でテストモデルを作成する
1. Claude Code に状態遷移図・シーケンス図を生成させる
2. VS Code のプレビューで確認し、テストケースの漏れ・エッジケースを視覚的に特定する

> **参考：** マネーフォワード事例「仕様書からテストケース自動生成でテスト時間70%削減」

---

### Phase 3：Git によるテスト成果物の管理

**目的：** テストケース作成と同時に Git 操作を習得する。実務での運用スタイルを体験する。

| 操作 | タイミング |
|---|---|
| `git init` / `git clone` | リポジトリ初期化・クローン |
| `git add` / `git commit` | テストケース作成のたびにコミット |
| `git branch` / `git checkout` | 機能ごとにブランチを切って作業 |
| `git diff` | テストケース修正時の差分確認 |
| `git push` | GitHub への共有 |

> **ポイント：** テストケースを1つ作るたびに commit する習慣をつける。

---

### Phase 4：不具合発見演習

**目的：** 作成したテストケースを使って実際にテストを実行し、不具合を発見・報告する。

1. `v1.8.1-FaRSeT-bug`（Port: 8083）を操作してテストを実行する
2. `atrs-1.8.1.SP1.RELEASE`（Port: 8082）と比較して期待動作を確認する
3. 発見した不具合をバグレポートとして記録する
4. `docs/ATRS サンプル不具合リスト - 講師用.xlsx` で答え合わせをする

---

### Phase 5：Linux コマンド基礎（AWS Ubuntu 環境）

**目的：** SSH でリモートサーバーに接続し、基本的な Linux コマンドを体験する。  
現場（MF）での Ubuntu / Linux 環境への対応力を養う。

**使用環境：** AWS t2.micro（メンターが事前構築済みの Ubuntu + nginx）

#### Step 1：SSH 接続
```bash
ssh <ユーザー名>@<AWS の IP アドレス>
```
- SSH キーの概念を理解する（ホスト・ポート・ユーザー）
- 接続が成功したことを確認する

#### Step 2：基本コマンドの練習
| コマンド | 操作内容 |
|---|---|
| `pwd` / `ls` / `cd` | 現在地確認・ディレクトリ移動 |
| `mkdir` / `touch` / `rm` | ファイル・ディレクトリの作成と削除 |
| `cat` / `less` | ファイルの中身を確認する |
| `grep` | ログやファイルから特定の文字列を探す |
| `ps` / `top` | プロセスの確認 |
| `sudo systemctl status nginx` | Web サーバーの動作確認 |

#### Step 3：ネットワークの確認
- AWS の IP アドレスにブラウザでアクセスし、nginx の画面が表示されることを確認する
- 「自分でサーバーを管理する」という感覚をつかむ

> **ポイント：** Linux そのものの習熟よりも「SSH でリモートサーバーに繋いで操作する」という体験が目的。

---

### Phase 6：Postman API テスト

**目的：** バックエンド API の動作を検証するテストを設計・実行する。  
AWS 環境での基礎練習から、ローカル ATRS への実践テストへ段階的にステップアップする。

#### Stage 1：AWS nginx を使った基礎練習（AWS 環境）

Postman の操作に慣れることを目的とし、シンプルな nginx を相手に練習する。

1. Postman で GET リクエストを `http://<AWS の IP アドレス>/` に送信する
2. レスポンスのステータスコード（200 OK）とボディを確認する
3. ヘッダー情報（Server, Content-Type）を読む練習をする

> **ここで身につけること：** リクエスト送信・レスポンス確認・ステータスコードの読み方

#### Stage 2：ローカル ATRS を使った実践（ローカル環境）

実際のテスト対象システム（ATRS）の API を Claude Code と壁打ちしながら検証する。

1. ATRS の API エンドポイントを特定する
2. Postman Agent Mode に正常系・異常系のリクエストを設計させる
3. 認証エラーなどの異常系を診断させる

> **参考：** マネーフォワード事例「OpenAPI 仕様から AI でテストケース自動生成」

---

### Phase 7：Playwright E2E テスト自動化

**目的：** テストケースをコードに落とし込み、自動テストの実行サイクルを体験する。

| エージェント | 役割 |
|---|---|
| **Planner** | アプリを探索し、テスト計画書（Markdown）を自動生成 |
| **Generator** | テスト計画書をもとに TypeScript のテストコードを自動生成・実行 |
| **Healer** | テスト失敗時にセレクタ変更等を自律的に特定・修復 |

> **参考：** マネーフォワード事例「AI エージェントが E2E テストの作成〜実行〜自己修復を自律的に繰り返す」

## トレーニング全体像

```
Phase 1：基礎知識（JSTQB / ISO 29119）
   ↓
Phase 2：テストケース作成 × コンテキストエンジニアリング  ← ★最優先
   ↓（並行）
Phase 3：Git によるバージョン管理
   ↓
Phase 4：不具合発見演習（ATRS で実践）
   ↓
Phase 5：Linux コマンド基礎（AWS Ubuntu）
   ↓
Phase 6：Postman API テスト
         Stage 1：AWS nginx で基礎練習
         Stage 2：ローカル ATRS で実践
   ↓
Phase 7：Playwright E2E テスト自動化
```

## 短縮版（6日間）の優先順位

参画前倒し（5/15 アサイン）の場合、以下の優先順位で進める。

| 優先度 | フェーズ | 目安日数 |
|---|---|---|
| 最優先 | Phase 2：テストケース設計 × Claude Code | 2〜3日 |
| 並行 | Phase 3：Git | Phase 2 と同時進行 |
| 高 | Phase 5：Linux コマンド基礎（AWS） | 0.5日 |
| 高 | Phase 6：Postman（Stage 1 → Stage 2） | 1〜1.5日 |
| 中 | Phase 4：不具合発見演習 | 1日 |
| 余裕があれば | Phase 7：Playwright | 時間次第 |
| 自習推奨 | Phase 1：基礎知識（ISO 29119） | 業務時間外 |
